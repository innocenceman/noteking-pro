# 第6集: 初始化项目

## 第 6 集：初始化项目

### 本集定位

本集是 MiniMind 项目的正式起步，主题是“初始化项目”。在真正实现 Transformer、训练循环、数据处理和推理脚本之前，首先要确定一个清晰、可维护、可扩展的项目结构，并定义一个统一管理模型超参数的配置类：`ModelArgs`。

{IMAGE:1}

本集虽然时长只有 3 分 47 秒，但它解决的是后续所有代码的“地基”问题：

- 项目目录应该如何组织
- 模型代码、训练代码、数据、日志、权重如何分层
- 为什么需要配置类
- `ModelArgs` 中应包含哪些关键参数
- 如何为后续手写大模型代码预留扩展空间

{IMPORTANT}本集核心：先搭好工程骨架，再写模型代码。项目结构决定了后续开发、调试、训练和复现实验的效率。{/IMPORTANT}

本节小结：本集不是在讲复杂算法，而是在为 MiniMind 项目建立工程基础。

## 一、为什么要先设计项目结构

### 1. 项目结构是大模型工程的入口

在从零实现一个大语言模型时，很多初学者容易直接新建一个 `main.py`，然后把模型、训练、数据加载、推理逻辑全部写在一起。这样短期看起来简单，但随着代码量增加，很快会出现问题：

- 模型结构难以复用
- 训练和推理逻辑互相耦合
- 参数散落在多个文件中
- 保存权重、日志、数据集时路径混乱
- 实验难以复现
- 后续添加 tokenizer、预训练、SFT、推理服务时难以扩展

{IMAGE:8}

对于 MiniMind 这类教学项目，虽然目标模型规模较小，但设计思路应该贴近真实大模型工程。也就是说，即使当前只训练一个小模型，也应该按照“可持续迭代”的方式组织代码。

{KNOWLEDGE}工程化的大模型项目通常会将“模型定义”“训练入口”“数据处理”“配置参数”“推理脚本”“权重文件”“日志输出”分开管理。这样做不是为了复杂，而是为了降低后续维护成本。{/KNOWLEDGE}

本节小结：项目结构不是形式主义，而是为了让后续模型开发、训练和调试更清晰。

## 二、MiniMind 项目的基础目录设计

### 1. 常见目录划分

本集的重点之一，是建立 MiniMind 的基础工程目录。一个简化但合理的结构可以设计为：

```text
MiniMind/
├── model/
│   ├── __init__.py
│   └── model.py
├── train/
│   └── pretrain.py
├── dataset/
│   └── ...
├── out/
│   └── ...
├── logs/
│   └── ...
├── scripts/
│   └── ...
└── README.md
```

{IMAGE:2}

其中，每个目录承担不同职责：

- `model/`：存放模型结构代码，例如 Transformer、Attention、MLP、RMSNorm 等
- `train/`：存放训练脚本，例如预训练、微调、分布式训练入口
- `dataset/`：存放数据或数据处理相关文件
- `out/`：保存模型权重、checkpoint、训练产物
- `logs/`：保存训练日志、loss 曲线记录等
- `scripts/`：存放辅助脚本，例如数据清洗、格式转换、测试命令
- `README.md`：记录项目说明、运行方式、依赖和实验结果

{IMAGE:3}

### 2. 为什么模型代码单独放在 `model/`

模型结构是整个项目最核心、最稳定、最需要复用的部分。训练脚本会调用模型，推理脚本也会调用模型，后续如果加入导出、量化或部署逻辑，仍然需要调用模型定义。

因此，模型代码不应该写死在训练脚本里，而应该独立成模块。

```python
# model/model.py

import torch
import torch.nn as nn

class MiniMindModel(nn.Module):
    def __init__(self, args):
        super().__init__()
        self.args = args
        # 后续会在这里定义 embedding、Transformer blocks、输出层等

    def forward(self, input_ids):
        # 后续实现前向传播逻辑
        pass
```

这种设计让训练脚本可以简单地导入模型：

```python
from model.model import MiniMindModel

model = MiniMindModel(args)
```

{WARNING}易错点：不要把模型定义、训练循环和数据处理全部堆在一个文件里。这样一开始写得快，但后面几乎无法维护。{/WARNING}

本节小结：合理的目录结构能让模型、训练、数据和输出结果各司其职。

## 三、配置类 ModelArgs 的作用

### 1. 为什么需要配置类

在大模型代码中，会出现大量超参数，例如：

- 词表大小
- 隐藏层维度
- Transformer 层数
- 注意力头数
- 最大序列长度
- dropout 概率
- 是否使用 bias
- 归一化层 epsilon
- 设备类型
- batch size
- 学习率

如果这些参数直接散落在代码中，例如：

```python
dim = 512
n_layers = 8
n_heads = 8
max_seq_len = 512
vocab_size = 6400
```

那么后续修改模型规模时，需要到处查找和修改，非常容易出错。

更好的方式是用一个配置类统一管理模型结构参数。

{IMAGE:4}

```python
from dataclasses import dataclass

@dataclass
class ModelArgs:
    dim: int = 512
    n_layers: int = 8
    n_heads: int = 8
    n_kv_heads: int = 8
    vocab_size: int = 6400
    hidden_dim: int | None = None
    multiple_of: int = 64
    norm_eps: float = 1e-5
    max_seq_len: int = 512
    dropout: float = 0.0
```

{IMPORTANT}ModelArgs 的本质是“模型结构说明书”。它把模型需要的核心超参数集中到一个地方，使模型构建、训练、实验记录和复现都更方便。{/IMPORTANT}

本节小结：`ModelArgs` 让模型参数集中管理，是后续实现 MiniMind 的核心入口之一。

## 四、ModelArgs 中的关键字段

### 1. `dim`：模型隐藏维度

`dim` 表示 token embedding 和 Transformer 内部隐藏状态的维度。

如果输入序列长度为 $T$，batch size 为 $B$，隐藏维度为 $D$，那么 embedding 后的张量形状通常是：

$$
X \in \mathbb{R}^{B \times T \times D}
$$

其中：

- $B$：batch size
- $T$：序列长度
- $D$：隐藏维度，即 `dim`

例如：

```python
dim = 512
```

表示每个 token 会被映射成一个 512 维向量。

{IMAGE:5}

维度越大，模型表达能力通常越强，但计算量和显存占用也会增加。

粗略来看，矩阵乘法的计算量常与 $D^2$ 相关，因此隐藏维度翻倍，计算成本可能显著增加。

本节小结：`dim` 决定模型内部表示空间大小，是模型规模的关键参数。

## 五、Transformer 层数与注意力头数

### 1. `n_layers`：网络深度

`n_layers` 表示 Transformer Block 的堆叠层数。

```python
n_layers: int = 8
```

如果 `n_layers = 8`，表示模型中会堆叠 8 个 Transformer Block。

从表示能力角度看，层数越多，模型越深，能够学习更复杂的层级特征。但层数增加也会带来：

- 训练时间增加
- 显存占用增加
- 梯度传播更困难
- 对数据规模和训练技巧要求更高

{IMAGE:9}

### 2. `n_heads`：注意力头数

`n_heads` 表示多头注意力中的 head 数量。

在标准多头注意力中，隐藏维度 `dim` 会被拆分到多个 head 上：

$$
head\_dim = \frac{dim}{n\_heads}
$$

例如：

```python
dim = 512
n_heads = 8
```

则：

$$
head\_dim = \frac{512}{8} = 64
$$

也就是说，每个注意力头处理 64 维特征。

{WARNING}易错点：`dim` 必须能够被 `n_heads` 整除，否则每个 head 的维度无法均分，后续 reshape 时会报错。{/WARNING}

可以在模型初始化时加入检查：

```python
assert args.dim % args.n_heads == 0, "dim 必须能被 n_heads 整除"
head_dim = args.dim // args.n_heads
```

本节小结：`n_layers` 控制深度，`n_heads` 控制多头注意力的并行子空间划分。

## 六、KV Heads 与注意力优化

### 1. `n_kv_heads` 的意义

`n_kv_heads` 表示 Key 和 Value 的头数。它通常用于支持 MHA、MQA 或 GQA 等不同注意力结构。

```python
n_kv_heads: int = 8
```

当：

```python
n_kv_heads == n_heads
```

时，通常对应标准 Multi-Head Attention。

如果：

```python
n_kv_heads < n_heads
```

则可能对应 Grouped-Query Attention，也就是多个 Query head 共享一组 Key/Value head。

{IMAGE:10}

### 2. 为什么要区分 Q heads 和 KV heads

在自回归语言模型推理时，Key 和 Value 通常会被缓存起来，即 KV Cache。序列越长，缓存越大。

KV Cache 的规模大致与以下因素相关：

$$
CacheSize \propto n\_layers \times seq\_len \times n\_kv\_heads \times head\_dim
$$

因此，减少 `n_kv_heads` 可以降低推理时 KV Cache 的显存占用。

{KNOWLEDGE}很多现代大模型会使用 GQA 来平衡效果与推理效率。它不像标准 MHA 那样每个 Query head 都有独立 KV，也不像 MQA 那样所有 Query head 共享同一组 KV，而是采用分组共享。{/KNOWLEDGE}

本节小结：`n_kv_heads` 是为后续支持更高效的注意力实现预留的重要参数。

## 七、词表大小与最大序列长度

### 1. `vocab_size`：词表大小

`vocab_size` 表示 tokenizer 中 token 的总数量。

```python
vocab_size: int = 6400
```

语言模型的输入通常是 token id，例如：

```python
input_ids = [15, 203, 91, 8]
```

这些 id 会通过 embedding 层映射成向量：

```python
self.tok_embeddings = nn.Embedding(args.vocab_size, args.dim)
```

如果 `vocab_size = 6400`，`dim = 512`，则 embedding 参数量为：

$$
6400 \times 512 = 3,276,800
$$

{IMAGE:11}

也就是说，仅 token embedding 层就有约 327 万个参数。

### 2. `max_seq_len`：最大上下文长度

`max_seq_len` 表示模型一次最多能处理的 token 数。

```python
max_seq_len: int = 512
```

如果输入超过这个长度，就需要截断、滑窗，或者扩展位置编码能力。

对训练而言，序列长度越长，注意力计算越贵。标准自注意力的复杂度约为：

$$
O(T^2 \cdot D)
$$

其中：

- $T$ 是序列长度
- $D$ 是隐藏维度

这说明序列长度翻倍，注意力矩阵规模会接近变为 4 倍。

{WARNING}易错点：`max_seq_len` 不只是一个数据处理参数，它还会影响位置编码、注意力 mask、KV Cache 和显存开销。{/WARNING}

本节小结：`vocab_size` 决定输入输出空间大小，`max_seq_len` 决定模型可处理的上下文长度。

## 八、MLP 隐藏层维度与 multiple_of

### 1. `hidden_dim` 的作用

Transformer Block 中除了 Attention，还有前馈神经网络，也就是 MLP 或 FFN。

通常 MLP 会先把隐藏维度放大，再投影回来：

$$
D \rightarrow H \rightarrow D
$$

其中：

- $D$ 是 `dim`
- $H$ 是 `hidden_dim`

例如：

```python
dim = 512
hidden_dim = 2048
```

则 MLP 的中间层维度是隐藏维度的 4 倍。

{IMAGE:12}

如果 `hidden_dim` 设置为 `None`，可以在模型内部根据 `dim` 自动计算。

```python
if args.hidden_dim is None:
    hidden_dim = 4 * args.dim
else:
    hidden_dim = args.hidden_dim
```

### 2. `multiple_of` 的作用

`multiple_of` 通常用于让 MLP 的隐藏层维度对齐到某个倍数，例如 64 或 256。

```python
multiple_of: int = 64
```

对齐后的维度更适合 GPU 计算，也有利于与一些高性能实现保持一致。

一个常见的对齐函数如下：

```python
def find_multiple(value: int, multiple_of: int) -> int:
    """将 value 向上对齐到 multiple_of 的整数倍"""
    return multiple_of * ((value + multiple_of - 1) // multiple_of)
```

例如：

$$
find\_multiple(2000, 64) = 2048
$$

本节小结：`hidden_dim` 决定 FFN 容量，`multiple_of` 用于让维度更适合硬件计算。

## 九、归一化参数与 Dropout

### 1. `norm_eps`

`norm_eps` 是归一化层中的数值稳定项。

```python
norm_eps: float = 1e-5
```

在 RMSNorm 或 LayerNorm 中，通常会出现类似计算：

$$
y = \frac{x}{\sqrt{mean(x^2) + \epsilon}}
$$

这里的 $\epsilon$ 就是 `norm_eps`，用于防止分母为 0 或过小导致数值不稳定。

{IMAGE:13}

```python
class RMSNorm(nn.Module):
    def __init__(self, dim: int, eps: float):
        super().__init__()
        self.eps = eps
        self.weight = nn.Parameter(torch.ones(dim))

    def forward(self, x):
        norm_x = x * torch.rsqrt(x.pow(2).mean(-1, keepdim=True) + self.eps)
        return self.weight * norm_x
```

### 2. `dropout`

`dropout` 用于训练阶段随机丢弃部分神经元，降低过拟合风险。

```python
dropout: float = 0.0
```

对于小模型或教学模型，初始设置为 0 是常见选择，因为这样可以减少变量，更方便观察模型是否正常收敛。

如果后续模型过拟合，可以再增加 dropout。

{KNOWLEDGE}Dropout 通常只在训练阶段生效，推理阶段会自动关闭。因此在 PyTorch 中要正确使用 `model.train()` 和 `model.eval()`。{/KNOWLEDGE}

本节小结：`norm_eps` 保证数值稳定，`dropout` 是常用正则化手段。

## 十、用 dataclass 定义 ModelArgs

### 1. dataclass 的优势

Python 的 `dataclass` 很适合定义配置类。它可以自动生成初始化方法，让代码更简洁。

```python
from dataclasses import dataclass

@dataclass
class ModelArgs:
    dim: int = 512
    n_layers: int = 8
    n_heads: int = 8
    n_kv_heads: int = 8
    vocab_size: int = 6400
    hidden_dim: int | None = None
    multiple_of: int = 64
    norm_eps: float = 1e-5
    max_seq_len: int = 512
    dropout: float = 0.0
```

{IMAGE:14}

创建配置对象：

```python
args = ModelArgs()
print(args.dim)        # 512
print(args.n_layers)   # 8
```

也可以覆盖默认参数：

```python
args = ModelArgs(
    dim=768,
    n_layers=12,
    n_heads=12,
    max_seq_len=1024
)
```

这样就可以快速切换不同规模的模型。

### 2. 配置类与实验复现

假设一次实验使用如下参数：

```python
args = ModelArgs(
    dim=512,
    n_layers=8,
    n_heads=8,
    vocab_size=6400,
    max_seq_len=512
)
```

如果训练结果不错，就应该把这组参数保存下来。因为模型权重本身并不能完整说明模型结构，加载权重时必须知道对应的结构参数。

{WARNING}易错点：保存 checkpoint 时不要只保存 `model.state_dict()`，还应保存模型配置。否则后续可能不知道用什么结构加载权重。{/WARNING}

示例：

```python
checkpoint = {
    "model": model.state_dict(),
    "model_args": args.__dict__,
    "step": step,
}
```

本节小结：`dataclass` 让配置定义清晰，配置对象也便于保存和复现实验。

## 十一、从配置到模型初始化

### 1. 配置如何进入模型

`ModelArgs` 的真正价值，是作为模型初始化的统一入口。

```python
class MiniMindModel(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()
        self.args = args

        self.tok_embeddings = nn.Embedding(
            args.vocab_size,
            args.dim
        )

        self.dropout = nn.Dropout(args.dropout)

        # 后续会加入 Transformer blocks
        self.layers = nn.ModuleList([
            # TransformerBlock(args) for _ in range(args.n_layers)
        ])

        self.norm = RMSNorm(args.dim, args.norm_eps)
        self.output = nn.Linear(args.dim, args.vocab_size, bias=False)
```

{IMAGE:15}

这样做的好处是：模型内部不需要写死任何结构参数，而是全部从 `args` 读取。

### 2. 参数量的粗略估计

通过 `ModelArgs`，我们可以估算模型参数规模。

例如 embedding 层参数量：

$$
Params_{emb} = vocab\_size \times dim
$$

输出层参数量：

$$
Params_{out} = dim \times vocab\_size
$$

如果 embedding 和输出层不共享权重，那么这两部分参数量相同。

当：

$$
vocab\_size = 6400,\quad dim = 512
$$

则：

$$
Params_{emb} = 6400 \times 512 = 3,276,800
$$

$$
Params_{out} = 512 \times 6400 = 3,276,800
$$

两者合计约 655 万参数。

{IMAGE:16}

本节小结：`ModelArgs` 是模型搭建的参数源，也可以帮助估算模型规模。

## 十二、项目初始化时应注意的问题

### 1. 保持文件职责单一

一个文件最好只做一类事情。例如：

- `model.py` 负责模型结构
- `pretrain.py` 负责预训练流程
- `tokenizer.py` 负责分词器相关逻辑
- `dataset.py` 负责数据读取和 batch 构造
- `config.py` 或 `model.py` 中定义配置类

{IMAGE:17}

这样做可以让调试路径更清晰。比如 loss 不下降时，可以分别检查：

- 数据是否正确
- token id 是否越界
- 模型 forward 是否输出正确形状
- loss 计算是否对齐
- optimizer 是否正常更新参数

### 2. 初始化阶段不要过早复杂化

虽然项目结构很重要，但也不应该一开始就引入过多复杂机制。例如：

- 暂时不需要复杂配置系统
- 暂时不需要大型训练框架
- 暂时不需要抽象过多基类
- 暂时不需要过度封装日志系统

MiniMind 的目标是从零手敲理解大模型，因此项目结构要“清晰但不过度工程化”。

{IMPORTANT}好的项目初始化不是把目录做得越复杂越好，而是让后续每一步都有明确位置。{/IMPORTANT}

本节小结：初始化阶段要兼顾清晰性和简洁性，避免把教学项目变成框架项目。

## 十三、本集知识点串联

### 1. 从目录到配置再到模型

本集内容可以串成一条主线：

```text
项目目录设计
    ↓
模型代码独立
    ↓
定义 ModelArgs
    ↓
用配置初始化模型
    ↓
为训练、保存、加载和推理打基础
```

{IMAGE:18}

这一集虽然没有真正展开 Transformer 的完整实现，但已经为后续内容做好准备。接下来实现 Attention、MLP、RMSNorm、位置编码、forward 逻辑时，都可以围绕 `ModelArgs` 逐步展开。

### 2. 一个完整的初始化示例

```python
from dataclasses import dataclass
import torch
import torch.nn as nn

@dataclass
class ModelArgs:
    # 模型隐藏维度
    dim: int = 512

    # Transformer Block 层数
    n_layers: int = 8

    # Query 注意力头数
    n_heads: int = 8

    # Key/Value 注意力头数
    n_kv_heads: int = 8

    # tokenizer 词表大小
    vocab_size: int = 6400

    # FFN 中间层维度；None 表示后续自动计算
    hidden_dim: int | None = None

    # FFN 维度对齐倍数
    multiple_of: int = 64

    # RMSNorm / LayerNorm 的稳定项
    norm_eps: float = 1e-5

    # 最大上下文长度
    max_seq_len: int = 512

    # dropout 概率
    dropout: float = 0.0


class MiniMindModel(nn.Module):
    def __init__(self, args: ModelArgs):
        super().__init__()

        assert args.dim % args.n_heads == 0, "dim 必须能被 n_heads 整除"

        self.args = args
        self.head_dim = args.dim // args.n_heads

        self.tok_embeddings = nn.Embedding(args.vocab_size, args.dim)
        self.dropout = nn.Dropout(args.dropout)

        # 后续课程会继续补充 Transformer blocks
        self.layers = nn.ModuleList([])

        self.output = nn.Linear(args.dim, args.vocab_size, bias=False)

    def forward(self, input_ids):
        x = self.tok_embeddings(input_ids)
        x = self.dropout(x)

        # 后续会依次通过 Transformer blocks
        logits = self.output(x)
        return logits
```

本节小结：项目初始化完成后，后续模型实现就可以围绕统一配置逐步推进。

## 十四、从教学项目到真实大模型工程

### 1. MiniMind 的结构设计价值

MiniMind 虽然是小模型，但它使用的很多工程思想和真实大模型项目是一致的：

- 配置驱动模型结构
- 模型定义独立于训练流程
- 权重输出独立管理
- 数据处理与训练解耦
- 保留日志和实验结果
- 便于后续支持预训练、微调和推理

{IMAGE:20}

真实工程中，类似 `ModelArgs` 的配置可能会扩展成更完整的配置系统，包括：

- `TrainArgs`
- `DataArgs`
- `OptimizerArgs`
- `TokenizerArgs`
- `GenerationArgs`

但在当前阶段，优先掌握 `ModelArgs` 已经足够。

### 2. 后续可能扩展的配置

例如训练配置可以单独定义：

```python
@dataclass
class TrainArgs:
    batch_size: int = 32
    learning_rate: float = 3e-4
    max_iters: int = 10000
    warmup_iters: int = 100
    weight_decay: float = 0.1
    grad_clip: float = 1.0
```

推理配置也可以单独定义：

```python
@dataclass
class GenerationArgs:
    max_new_tokens: int = 100
    temperature: float = 0.8
    top_k: int = 50
    top_p: float = 0.9
```

这样可以避免把模型结构参数、训练参数和生成参数混在一起。

{WARNING}易错点：`ModelArgs` 只应该描述模型结构本身，不建议把学习率、batch size、训练轮数等训练参数塞进去。{/WARNING}

本节小结：MiniMind 的初始化方式可以自然扩展到更完整的大模型工程。

## 十五、课堂总结

{IMAGE:21}

本集完成了 MiniMind 项目的基础初始化。核心不是写了多少模型代码，而是明确了后续开发的组织方式。

### Key Takeaways

1. 项目结构要先于模型实现设计好，否则后续训练、推理、保存权重都会变得混乱。
2. `model/`、`train/`、`dataset/`、`out/`、`logs/` 等目录应职责分明。
3. `ModelArgs` 是模型结构配置类，用于集中管理隐藏维度、层数、注意力头数、词表大小、最大序列长度等参数。
4. `dim`、`n_layers`、`n_heads`、`vocab_size`、`max_seq_len` 是决定模型规模和计算成本的关键参数。
5. 多头注意力要求 $dim$ 能够被 $n\_heads$ 整除。
6. `n_kv_heads` 为后续实现 MHA、MQA、GQA 等注意力结构留下扩展空间。
7. 保存模型时应同时保存模型权重和模型配置，便于后续复现实验。
8. 初始化阶段应清晰但不过度工程化，避免在教学项目中过早引入复杂框架。
