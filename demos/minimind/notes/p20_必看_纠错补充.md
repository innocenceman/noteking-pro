## 一、为什么需要“纠错补充”

{IMAGE:1}

在大模型训练代码中，很多问题并不是 Python 语法错误，而是“数学意义正确性”的问题。例如：

- tensor 维度虽然能 broadcast，但语义不对；
- mask 能跑通，但遮挡范围不严谨；
- loss 能计算，但 label 对齐错位；
- 推理能生成文字，但采样策略没有正确生效；
- checkpoint 能保存，但恢复训练时 optimizer 或 scheduler 状态缺失。

这些问题往往不会直接触发异常，却会影响最终模型效果。

{KNOWLEDGE}深度学习代码的常见特点是：只要张量形状能对上，程序就可能继续运行。但“能运行”不等于“逻辑正确”。{/KNOWLEDGE}

在 MiniMind 这种从零实现的课程中，尤其要养成两种习惯：

1. 看维度：每一步输入输出的 shape 是否符合预期；
2. 看语义：这个 tensor 在数学上表示什么，是否与公式一致。

本节总结：大模型代码最危险的问题通常不是报错，而是“静默错误”，因此必须定期勘误和复盘。

## 二、Transformer 中最容易出错的维度问题

{IMAGE:2}

Transformer 的核心结构中，最容易写错的是 attention 部分。设输入为：

$$
X \in \mathbb{R}^{B \times T \times C}
$$

其中：

- $B$：batch size；
- $T$：序列长度；
- $C$：隐藏层维度，即 embedding dimension。

经过线性层得到：

$$
Q = XW_Q,\quad K = XW_K,\quad V = XW_V
$$

在多头注意力中，需要把 $C$ 拆成：

$$
C = H \times D
$$

其中：

- $H$：attention heads 数量；
- $D$：每个 head 的维度。

因此常见 reshape 过程是：

```python
# x: [B, T, C]
B, T, C = x.shape

q = self.wq(x)  # [B, T, C]
k = self.wk(x)  # [B, T, C]
v = self.wv(x)  # [B, T, C]

# 拆成多头: [B, T, H, D]
q = q.view(B, T, self.n_heads, self.head_dim)
k = k.view(B, T, self.n_heads, self.head_dim)
v = v.view(B, T, self.n_heads, self.head_dim)

# 调整维度到 attention 计算需要的格式: [B, H, T, D]
q = q.transpose(1, 2)
k = k.transpose(1, 2)
v = v.transpose(1, 2)
```

{WARNING}常见错误是只 reshape 不 transpose，或者 transpose 后忘记 contiguous，再直接 view，导致内存布局和逻辑维度不一致。{/WARNING}

如果后续需要把多头结果拼回去，推荐写法是：

```python
# out: [B, H, T, D]
out = out.transpose(1, 2).contiguous()  # [B, T, H, D]
out = out.view(B, T, C)                 # [B, T, C]
```

本节总结：attention 的维度变换必须同时检查 shape 和语义，尤其注意 `view`、`transpose`、`contiguous` 的配合。

## 三、Attention 缩放因子与 Mask 的补充

{IMAGE:3}

Scaled Dot-Product Attention 的公式为：

$$
\text{Attention}(Q,K,V)=\text{softmax}\left(\frac{QK^T}{\sqrt{d_k}} + M\right)V
$$

其中 $d_k$ 是每个 head 的维度，而不是总 hidden size。

如果误用 $C$ 作为缩放因子，会导致 attention logits 被过度缩小：

$$
\frac{QK^T}{\sqrt{C}} \neq \frac{QK^T}{\sqrt{D}}
$$

正确实现通常是：

```python
import math
import torch
import torch.nn.functional as F

scores = q @ k.transpose(-2, -1) / math.sqrt(self.head_dim)
```

对于自回归语言模型，当前位置只能看到当前位置及其之前的 token，不能看到未来 token。因此需要 causal mask。

```python
# scores: [B, H, T, T]
mask = torch.tril(torch.ones(T, T, device=x.device)).view(1, 1, T, T)

# 将未来位置置为一个极小值，使 softmax 后接近 0
scores = scores.masked_fill(mask == 0, float("-inf"))

att = F.softmax(scores, dim=-1)
out = att @ v
```

{IMPORTANT}自回归模型的 mask 不是为了提升速度，而是为了保证训练目标与推理过程一致：模型预测第 $t$ 个 token 时不能提前看到第 $t+1$ 个 token。{/IMPORTANT}

本节总结：attention 缩放必须使用 head_dim，causal mask 必须严格阻止模型看到未来 token。

## 四、Loss 对齐：输入与标签的错位关系

{IMAGE:13}

语言模型训练的目标是：给定前面的 token，预测下一个 token。

如果原始 token 序列为：

$$
[x_0, x_1, x_2, x_3, x_4]
$$

那么训练时通常使用：

$$
\text{input} = [x_0, x_1, x_2, x_3]
$$

$$
\text{target} = [x_1, x_2, x_3, x_4]
$$

也就是说，模型在位置 $t$ 的输出，应该和位置 $t+1$ 的真实 token 计算 loss。

```python
# batch: [B, T]
x = batch[:, :-1]   # 输入 token
y = batch[:, 1:]    # 目标 token

logits = model(x)   # [B, T-1, vocab_size]

loss = F.cross_entropy(
    logits.reshape(-1, logits.size(-1)),
    y.reshape(-1)
)
```

{WARNING}如果直接用同一个 batch 同时作为输入和标签，模型可能学成“复制当前 token”，而不是预测下一个 token。{/WARNING}

对于自回归语言模型，交叉熵损失为：

$$
\mathcal{L}=-\frac{1}{N}\sum_{i=1}^{N}\log p(y_i|x_{\le i})
$$

其中 $y_i$ 是当前位置应该预测的真实下一个 token。

本节总结：语言模型训练的关键是 next-token prediction，输入和标签必须错一位。

## 五、Tokenizer 与特殊符号的补充说明

{IMAGE:14}

在 MiniMind 这种小模型中，tokenizer 的设计会直接影响训练效果。常见特殊 token 包括：

- `<pad>`：填充；
- `<bos>`：序列开始；
- `<eos>`：序列结束；
- `<unk>`：未知字符或未知词；
- `<s>` / `</s>`：对话或句子边界。

如果训练数据中加入了 `<eos>`，模型才能学会“什么时候结束”。否则推理时模型往往只能靠最大长度截断。

```python
text = "你好，MiniMind"
ids = tokenizer.encode(text)

# 一种常见做法：手动添加开始和结束标记
ids = [tokenizer.bos_id] + ids + [tokenizer.eos_id]
```

{KNOWLEDGE}特殊 token 不是装饰，它们是模型学习文本边界、填充位置、对话结构的重要信号。{/KNOWLEDGE}

如果 batch 内序列长度不同，需要 padding，并在 loss 中忽略 pad 位置：

```python
loss = F.cross_entropy(
    logits.reshape(-1, logits.size(-1)),
    targets.reshape(-1),
    ignore_index=tokenizer.pad_id
)
```

本节总结：tokenizer 的特殊符号会影响训练目标，尤其是 `<eos>` 和 `<pad>` 的处理不能随意省略。

## 六、训练循环中的常见勘误点

{IMAGE:15}

训练循环看似简单，但容易出现多个隐性问题。

### 1. 梯度清零位置

标准训练流程为：

```python
for step, batch in enumerate(train_loader):
    optimizer.zero_grad(set_to_none=True)

    logits = model(batch["input_ids"])
    loss = criterion(logits, batch["labels"])

    loss.backward()
    optimizer.step()
```

如果忘记 `optimizer.zero_grad()`，梯度会在多个 batch 之间累积。除非明确做 gradient accumulation，否则这通常是错误的。

### 2. train / eval 模式切换

训练时：

```python
model.train()
```

验证或推理时：

```python
model.eval()
```

`Dropout`、`LayerNorm`、某些缓存逻辑都可能受到模式影响。虽然 LayerNorm 本身 train/eval 行为基本一致，但 Dropout 一定不同。

```python
model.eval()
with torch.no_grad():
    logits = model(input_ids)
```

{WARNING}推理时如果忘记 `model.eval()`，Dropout 仍然生效，同一个输入可能得到不稳定输出。{/WARNING}

### 3. mixed precision 与 scaler

如果使用 AMP，需要注意：

```python
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

而不是在 scaler 管理之外直接 `optimizer.step()`。

本节总结：训练循环中的梯度清零、模式切换、AMP 步骤都属于基础但高频出错点。

## 七、学习率与训练稳定性的补充

{IMAGE:16}

学习率是小模型训练中最敏感的超参数之一。过大时 loss 可能震荡甚至发散；过小时训练非常慢。

常见学习率调度包括 warmup + cosine decay：

$$
lr(t)=lr_{\max}\cdot \frac{t}{T_{warmup}},\quad t<T_{warmup}
$$

warmup 后使用余弦衰减：

$$
lr(t)=lr_{\min}+\frac{1}{2}(lr_{\max}-lr_{\min})
\left(1+\cos\frac{\pi(t-T_{warmup})}{T-T_{warmup}}\right)
$$

示例代码：

```python
def get_lr(step, max_steps, warmup_steps, max_lr, min_lr):
    if step < warmup_steps:
        return max_lr * step / warmup_steps

    progress = (step - warmup_steps) / (max_steps - warmup_steps)
    cosine = 0.5 * (1.0 + math.cos(math.pi * progress))
    return min_lr + cosine * (max_lr - min_lr)
```

{IMPORTANT}warmup 的作用是让模型在训练初期逐渐适应梯度尺度，避免一开始参数更新过猛。{/IMPORTANT}

本节总结：学习率策略直接影响训练稳定性，warmup + cosine 是小型 Transformer 常用且稳健的选择。

## 八、保存与加载 Checkpoint 的补充

{IMAGE:17}

只保存模型参数有时不够。若要恢复训练，应同时保存：

- model state；
- optimizer state；
- scheduler state；
- 当前 step 或 epoch；
- scaler state，如果使用 AMP；
- config 信息。

```python
ckpt = {
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
    "scheduler": scheduler.state_dict() if scheduler else None,
    "step": step,
    "config": config,
}

torch.save(ckpt, "checkpoint.pt")
```

加载时：

```python
ckpt = torch.load("checkpoint.pt", map_location=device)

model.load_state_dict(ckpt["model"])
optimizer.load_state_dict(ckpt["optimizer"])

if scheduler and ckpt["scheduler"] is not None:
    scheduler.load_state_dict(ckpt["scheduler"])

start_step = ckpt["step"]
```

{WARNING}如果只加载模型参数而不加载 optimizer 状态，恢复训练时动量、二阶矩估计等都会丢失，训练曲线可能出现明显跳变。{/WARNING}

本节总结：checkpoint 不只是模型权重文件，完整恢复训练需要保存训练状态。

## 九、推理阶段的采样策略补充

{IMAGE:18}

推理生成时，模型输出的是 vocabulary 上的 logits：

$$
logits \in \mathbb{R}^{V}
$$

需要把 logits 转成概率分布：

$$
p_i = \frac{e^{z_i/T}}{\sum_j e^{z_j/T}}
$$

其中 $T$ 是 temperature。注意常见写法是：

```python
logits = logits / temperature
probs = torch.softmax(logits, dim=-1)
```

temperature 的影响：

- $T < 1$：分布更尖锐，输出更确定；
- $T = 1$：保持原始分布；
- $T > 1$：分布更平滑，输出更多样。

### Top-k 采样

```python
def top_k_sample(logits, k):
    values, indices = torch.topk(logits, k)

    filtered = torch.full_like(logits, float("-inf"))
    filtered.scatter_(dim=-1, index=indices, src=values)

    probs = torch.softmax(filtered, dim=-1)
    next_id = torch.multinomial(probs, num_samples=1)
    return next_id
```

{IMPORTANT}贪心解码不一定最好。对于开放式文本生成，适当使用 temperature、top-k 或 top-p 可以提升多样性。{/IMPORTANT}

本节总结：推理不是简单 argmax，采样策略会显著影响生成文本的风格和质量。

## 十、KV Cache 与推理效率

{IMAGE:19}

在自回归生成中，每次生成一个新 token。如果每一步都把完整序列重新送入模型，会重复计算历史 token 的 $K,V$。

没有 cache 时，第 $t$ 步需要重新计算：

$$
K_{1:t}, V_{1:t}
$$

有 KV cache 时，只需要计算新 token 对应的：

$$
K_t, V_t
$$

然后与历史 cache 拼接。

简化伪代码：

```python
past_k, past_v = None, None

for _ in range(max_new_tokens):
    logits, past_k, past_v = model(
        input_ids[:, -1:],
        past_k=past_k,
        past_v=past_v,
        use_cache=True
    )

    next_id = sample(logits[:, -1, :])
    input_ids = torch.cat([input_ids, next_id], dim=1)
```

{KNOWLEDGE}KV Cache 不改变模型输出的数学含义，它只是避免重复计算历史 token 的 key/value，从而提升推理速度。{/KNOWLEDGE}

本节总结：KV cache 是推理优化手段，不是训练必需项；它提升速度，但也会占用额外显存。

## 十一、设备、dtype 与显存问题

{IMAGE:20}

常见设备错误包括：

- model 在 GPU，input 在 CPU；
- mask 在 CPU，scores 在 GPU；
- position_ids dtype 不对；
- 混合精度下某些张量仍是 fp32 或 fp16，不符合预期。

推荐在创建张量时显式指定 device：

```python
device = input_ids.device
T = input_ids.size(1)

mask = torch.tril(torch.ones(T, T, device=device))
```

如果使用 position embedding：

```python
pos = torch.arange(0, T, device=device).unsqueeze(0)
```

{WARNING}不要在 forward 中随意创建默认 CPU tensor。只要它参与 GPU tensor 运算，就可能触发 device mismatch。{/WARNING}

对于 dtype，一般遵循：

- token ids：`torch.long`；
- embedding 权重：float；
- logits：float；
- labels：`torch.long`；
- attention mask：bool 或 float mask。

本节总结：device 和 dtype 是训练脚本中最常见的运行时错误来源，应在创建张量时主动对齐。

## 十二、参数量估算与模型配置补充

{IMAGE:21}

MiniMind 作为小型语言模型，需要理解配置对参数量和速度的影响。核心配置包括：

- vocab_size；
- hidden_size；
- num_layers；
- num_heads；
- intermediate_size；
- max_seq_len。

Embedding 参数量约为：

$$
P_{emb}=V \times C
$$

其中 $V$ 是词表大小，$C$ 是隐藏维度。

单层 attention 中，Q/K/V/O 四个线性层参数量约为：

$$
P_{attn}=4C^2
$$

MLP 若使用两层结构，hidden expansion 为 $r$，参数量约为：

$$
P_{mlp}=2rC^2
$$

因此 Transformer 主体参数量可粗略估算为：

$$
P \approx V C + L(4C^2 + 2rC^2)
$$

其中 $L$ 是层数。

{IMPORTANT}小模型调参时，不要只盯着 layer 数。hidden size、词表大小、MLP expansion 都会显著影响参数量和显存。{/IMPORTANT}

本节总结：理解参数量公式可以帮助判断模型配置是否合理，避免盲目堆大模型。

## 十三、数据清洗与样本质量补充

{IMAGE:22}

语言模型的能力不仅取决于结构，也高度依赖数据质量。常见问题包括：

- 重复样本过多；
- 文本过短或无意义；
- 编码异常；
- 包含大量噪声符号；
- 训练语料与目标任务风格不一致。

简单的数据清洗示例：

```python
def clean_text(text):
    text = text.strip()

    # 过滤过短文本
    if len(text) < 5:
        return None

    # 过滤空白占比过高的文本
    blank_ratio = sum(ch.isspace() for ch in text) / max(len(text), 1)
    if blank_ratio > 0.5:
        return None

    return text
```

{KNOWLEDGE}对小模型来说，数据质量的重要性往往高于数据规模。少量高质量数据通常比大量噪声数据更有效。{/KNOWLEDGE}

本节总结：训练效果不好时，不应只怀疑模型结构，也要检查数据质量和数据格式。

## 十四、评估方式与误判风险

{IMAGE:23}

只看训练 loss 不能完整判断模型能力。训练 loss 下降只能说明模型更好地拟合了训练集，不代表泛化能力一定提升。

常见评估方式包括：

- validation loss；
- perplexity；
- 固定 prompt 生成质量；
- 任务型问答测试；
- 人工检查语义连贯性。

困惑度定义为：

$$
PPL = e^{\mathcal{L}}
$$

其中 $\mathcal{L}$ 是平均交叉熵损失。

如果 loss 为 2.0：

$$
PPL=e^2 \approx 7.39
$$

含义是：模型平均在约 7.39 个候选 token 中困惑。

{WARNING}生成几个看起来不错的例子，不能证明模型整体能力强。必须结合验证集指标和固定测试集进行比较。{/WARNING}

本节总结：评估模型要结合 loss、PPL 和固定样例，避免被个别生成结果误导。

## 十五、代码风格与可维护性补充

{IMAGE:24}

从零实现模型时，初期可以把代码写得直观，但随着功能增加，需要逐步提高结构清晰度。建议将项目拆分为：

- `model.py`：模型结构；
- `train.py`：训练入口；
- `dataset.py`：数据读取；
- `tokenizer.py`：分词器；
- `generate.py`：推理生成；
- `config.py`：配置管理。

一个清晰的 config 示例：

```python
from dataclasses import dataclass

@dataclass
class MiniMindConfig:
    vocab_size: int = 6400
    hidden_size: int = 512
    num_layers: int = 8
    num_heads: int = 8
    max_seq_len: int = 512
    dropout: float = 0.1
```

{IMPORTANT}可维护性不是“大项目才需要”的东西。训练代码一旦出现多个实验版本，没有清晰配置就很难复现。{/IMPORTANT}

本节总结：代码结构清楚、配置集中管理，是后续调参和复现实验的基础。

## 十六、勘误后的自检清单

{IMAGE:4}

完成本集纠错后，可以用以下清单检查自己的 MiniMind 实现：

### 模型结构检查

- attention 缩放因子是否使用 `head_dim`；
- Q/K/V reshape 和 transpose 是否正确；
- causal mask 是否正确遮挡未来 token；
- position embedding 是否与 sequence length 对齐；
- residual connection 是否放在正确位置；
- LayerNorm 顺序是否与设计一致。

### 训练逻辑检查

- input 和 target 是否错位；
- pad token 是否在 loss 中忽略；
- 是否正确清零梯度；
- 是否正确切换 `train()` / `eval()`；
- checkpoint 是否保存 optimizer 状态；
- 学习率调度是否按 step 更新。

### 推理逻辑检查

- 是否使用 `torch.no_grad()`；
- 是否正确截断上下文长度；
- temperature 是否避免为 0；
- top-k/top-p 是否在 softmax 前处理 logits；
- 是否遇到 `<eos>` 后停止生成。

{WARNING}很多训练失败不是由单一大错误导致，而是多个小细节叠加造成的。{/WARNING}

本节总结：勘误的价值在于建立自检体系，避免同类问题在后续实验中反复出现。

## 十七、示例：一个更稳健的生成函数

{IMAGE:5}

下面给出一个更稳健的自回归生成示例，整合了 eval 模式、no_grad、temperature、top-k 和 eos 停止逻辑：

```python
import torch
import torch.nn.functional as F

@torch.no_grad()
def generate(
    model,
    input_ids,
    max_new_tokens,
    eos_id=None,
    temperature=1.0,
    top_k=None,
):
    model.eval()

    for _ in range(max_new_tokens):
        # 如果序列超过模型最大长度，只保留最后一段上下文
        idx_cond = input_ids[:, -model.config.max_seq_len:]

        logits = model(idx_cond)          # [B, T, vocab_size]
        logits = logits[:, -1, :]         # 只取最后一个位置

        if temperature <= 0:
            # temperature 为 0 时退化为贪心搜索
            next_id = torch.argmax(logits, dim=-1, keepdim=True)
        else:
            logits = logits / temperature

            if top_k is not None:
                values, _ = torch.topk(logits, top_k)
                min_value = values[:, [-1]]
                logits = torch.where(
                    logits < min_value,
                    torch.full_like(logits, float("-inf")),
                    logits
                )

            probs = F.softmax(logits, dim=-1)
            next_id = torch.multinomial(probs, num_samples=1)

        input_ids = torch.cat([input_ids, next_id], dim=1)

        if eos_id is not None and torch.all(next_id == eos_id):
            break

    return input_ids
```

这段代码体现了几个关键点：

- 推理阶段关闭梯度；
- 每次只根据最后一个 token 的 logits 采样；
- temperature 为 0 时使用 argmax；
- top-k 在 softmax 前过滤；
- 支持遇到 eos 后提前停止。

本节总结：生成函数虽然短，但包含许多容易被忽略的边界条件。

## 十八、训练脚本中的推荐断言

{IMAGE:6}

为了尽早发现错误，可以在开发阶段加入断言：

```python
def check_batch(input_ids, labels, vocab_size):
    assert input_ids.dtype == torch.long
    assert labels.dtype == torch.long

    assert input_ids.shape == labels.shape

    assert input_ids.min() >= 0
    assert input_ids.max() < vocab_size

    # labels 如果有 ignore_index，则需要单独处理
    valid_labels = labels[labels >= 0]
    assert valid_labels.max() < vocab_size
```

attention 中也可以检查：

```python
assert C % self.n_heads == 0
assert q.shape == (B, self.n_heads, T, self.head_dim)
assert k.shape == (B, self.n_heads, T, self.head_dim)
assert v.shape == (B, self.n_heads, T, self.head_dim)
```

{KNOWLEDGE}断言不是最终产品代码的负担，而是开发阶段定位问题的高效工具。尤其在从零实现模型时，断言可以帮助快速锁定 shape 错误。{/KNOWLEDGE}

本节总结：用断言保护关键维度和取值范围，可以显著降低调试成本。

{IMAGE:25}

本集最重要的价值，是提醒学习者从“代码跑通”进入“代码可信”的阶段。

- 跑通：程序没有报错；
- 可训练：loss 能下降；
- 可信：每个模块的数学意义正确；
- 可复现：同样配置可以重复得到接近结果；
- 可扩展：后续能继续加入功能而不混乱。

{WARNING}只追求跑通，容易积累技术债。后续模型变大、数据变多、训练更久时，小错误会被放大成大成本。{/WARNING}

本节总结：从零实现大模型的真正收获，不只是写出一个模型，而是建立严谨的工程和实验习惯。
